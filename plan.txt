КОНТЕКСТ ПРОЕКТА (краткий аудит)
───────────────────────────────────────────────────────────────────────────────
- Тип: Flask-приложение (Python), запускается через api.py
- Архитектура:
    api.py                 — точка входа: загрузка .env, init БД, запуск
                             фоновых задач, запуск cloudflared-тоннеля,
                             запуск Flask на 0.0.0.0:$PORT
    freeapi/app.py         — фабрика Flask app
    freeapi/database.py    — инициализация SQLite + миграции
    freeapi/blueprints/*   — модульные роуты (auth, chat, community, keys,
                             reviews, support, admin, tg, notifications, ...)
    freeapi/repos/*        — слой работы с БД (репозитории)
    freeapi/scheduler.py   — фоновые задачи
    freeapi/tunnel.py      — менеджер cloudflared-тоннеля
    freeapi/agent.py       — опциональный AI-агент (Favorite)
    freeapi/tg_notify.py   — Telegram-уведомления о новом URL и упоминаниях
    static/                — фронт (index.html + JS/CSS модули)
    freeapi/migrations/    — SQL-миграции (15 файлов)
- Фронт: static/index.html + static/js/{core,ui,views} + static/css/main.css
- БД: SQLite, схема через миграции 001..015
- Подключение «наружу»: cloudflared (trycloudflare.com) + опц. tg-уведомления

[X] T8. ИСТИННЫЙ корень бага «вход вроде есть, а юзер — гость»
        ── Файл: freeapi/repos/users.py
        ── Симптом из лог-файла logi_(1)_1777151437234.txt:
              POST /api/auth/login → 500
              KeyError: 'password_hash' в auth_service.py:22
              check_password_hash(user['password_hash'], password)
              после этого: GET /api/keys → 401, юзер остаётся гостем,
              но JS-обвязка модалки уже закрыла её и переключила
              view на dashboard → визуально «вошёл, но Гость».
        ── Корень: в freeapi/repos/users.py было ДВА определения
                   функции get_user_by_username. Второе (внизу файла,
                   добавлено вместе с display_prefix) перезатирало
                   первое и селектило только id, username, display_prefix
                   — без password_hash. login_user() обращался
                   к user['password_hash'] и падал с KeyError.
        ── Решение:
              • оставлено ОДНО определение get_user_by_username
                с явным SELECT id, username, password_hash,
                display_prefix, created_at, last_login_at;
              • дубликат внизу файла удалён;
              • get_user_by_id заодно расширен на display_prefix
                (оставаясь лёгким — без password_hash и avatar).
        ── Эффект: POST /api/auth/login снова возвращает 200,
                   ставит session-cookie, /api/auth/me видит юзера,
                   /api/keys отдаёт 200, сайдбар рисует ник,
                   а не «Гость / Не авторизован».
        ── Статус: ВЫПОЛНЕНО

A2 — Статистика моделей показывает 0 req
────────────────────────────────────────
Статус: [✓] ВЫПОЛНЕНО
Сложность: малая (1 файл, ~4 строки добавить).

Проблема:
  «Топ модели» и «Доступные модели» показывают 0 req у всех моделей.
  В истории ключа реальные запросы видны.

РЕАЛЬНЫЙ корень (установлен при чтении кода):
  В freeapi/blueprints/chat_bp.py: update_model_stats() УЖЕ вызывается
  после каждого finish_request (строки 80, 99, 279, 306, 321).
  НО в freeapi/agent.py (AI-модератор): finish_request вызывается
  (строки 203, 210), а update_model_stats — НЕТ вообще.
  Значит, запросы через AI-агента (модерация отзывов) не учитываются в статистике.

  Вторая возможная причина: model_stats заполняется ТОЛЬКО для текущего месяца.
  При пустой БД или на старте месяца — строки вставляются через _seed_reference_data
  (INSERT OR IGNORE), но total_requests = 0. Запросы из chat_bp должны накапливаться.

  Дополнительная проверка: в _seed_reference_data() — row для каждой модели
  создаётся INSERT OR IGNORE, т.е. при уже существующей строке ничего не вставится.
  Если запросы уже делались, статистика должна быть ненулевой. Если нет — баг в agent.py.

Решение — добавить вызовы update_model_stats в agent.py:

  В freeapi/agent.py, функция _moderate(), блок try/except (≈строки 198-212):

  # Успешный ответ (строка ≈203):
  repo.finish_request(req_record['id'], 'ok', 'REQ_OK_001',
                      response_ms=int((_time.time() - _t0) * 1000))
  # ДОБАВИТЬ:
  try:
      from freeapi.repos.stats import update_model_stats
      update_model_stats(model, int((_time.time() - _t0) * 1000), ok=True)
  except Exception:
      pass

  # Ошибка (строка ≈210):
  repo.finish_request(req_record['id'], 'error', 'REQ_ERR_500',
                      response_ms=int((_time.time() - _t0) * 1000), ...)
  # ДОБАВИТЬ:
  try:
      from freeapi.repos.stats import update_model_stats
      update_model_stats(model, int((_time.time() - _t0) * 1000), ok=False)
  except Exception:
      pass

  Внимание: elapsed уже вычислен внутри блоков. Использовать сохранённое значение,
  не пересчитывать _time.time()-_t0 повторно (уже поздно). Лучше сохранить elapsed
  в переменную перед вызовом finish_request, как это сделано в chat_bp.py.

Push: «fix(stats): A2 — update_model_stats добавлен в agent.py».

A3 — Цветовая индикация response_ms в истории ключа
────────────────────────────────────────
Статус: [✓] ВЫПОЛНЕНО
Сложность: минимальная (1 JS-файл, 5 строк).

Проблема:
  В истории запросов (views/keys.js, функция showHistory, строка 305):
    '<div class="hist-ms">' + (r.response_ms != null ? r.response_ms + ' мс' : '—') + '</div>'
  Нет цветовой индикации — пользователь не видит, быстро или медленно.

Реальная задержка Gemini: 9–12 секунд — это норма для thinking-моделей.
Цвета помогают понять контекст.

Решение — в static/js/views/keys.js, строка 305, заменить на:
  var _ms = r.response_ms;
  var _msColor = _ms == null ? '#555' : (_ms < 4000 ? '#22c55e' : (_ms < 8000 ? '#eab308' : '#ef4444'));
  '<div class="hist-ms" style="color:' + _msColor + '">' +
    (_ms != null ? _ms + ' мс' : '—') +
  '</div>'

  Пороги:
    < 4000 мс  → зелёный #22c55e
    4000–8000  → жёлтый  #eab308
    > 8000 мс  → красный #ef4444

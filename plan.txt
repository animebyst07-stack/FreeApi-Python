A1 — Отступ сверху: «Новый ключ» и «Назад к дашборду»
────────────────────────────────────────
Статус: [ ] НЕ НАЧАТО

Проблема (видно на скриншотах):
  • Кнопка «+ Новый ключ» и заголовок «API КЛЮЧИ» прилипают к шапке.
  • Кнопка «Назад к дашборду» тоже вплотную к шапке.
  • Текущий .dash{padding-top:36px} → недостаточно, ощущается как 0 из-за

Решение:
  В static/css/main.css:
    .dash { padding: 52px 0 64px }          /* ← было 36px */

  Дополнительно — кнопка «Назад к дашборду»:
    В index.html найти элемент .back-btn (или аналог) и добавить ему
    margin-top или убедиться что он внутри .dash контейнера.
    Если он снаружи — перенести внутрь .dash.

Push: один коммит «fix(ui): A1 — увеличен padding-top для dash».

A2 — Статистика моделей показывает 0 req
────────────────────────────────────────
Статус: [ ] НЕ НАЧАТО

Проблема:
  В разделе «Топ модели» (главная) и «Доступные модели» (документация)
  у всех моделей стоит «0 req» / «ЗАПРОСОВ: 0», хотя в истории ключа
  видны реальные запросы.

Корень (предположение после анализа кода):
  • update_model_stats() вызывается из finish_request() в repos/keys.py?
    → Нужно проверить. Если НЕ вызывается — это и есть баг.
  • Возможно update_model_stats вообще не вызывается ни из одного места.
  • Модель в запросах — «gemini-3.0-flash», но в AI_MODELS id может
    отличаться — тогда UPDATE не совпадает по model_id.

Диагностика:
  1. grep -rn "update_model_stats" freeapi/ — найти все вызовы.
  2. Сравнить model['id'] из AI_MODELS с тем, что записывается в requests.model.
  3. Если вызовов нет → добавить вызов update_model_stats внутрь finish_request.
  4. Если id не совпадают → исправить маппинг.

Решение (ожидаемое):
  В freeapi/repos/keys.py в функции finish_request после UPDATE requests:
    repo.update_model_stats(model_id, response_ms, ok=(status == 'ok'))
  Но model_id нужно брать из запроса (сделать SELECT model FROM requests WHERE id=?).
  Либо передавать model_id параметром в finish_request (изменить сигнатуру и все вызовы).

Push: один коммит «fix(stats): A2 — update_model_stats теперь вызывается из finish_request».

A3 — Задержка 9–11 секунд в ответах
────────────────────────────────────────
Статус: [ ] НЕ НАЧАТО

Проблема:
  В истории ключа видно: 9967 мс, 10632 мс, 11700 мс.
  Это РЕАЛЬНАЯ задержка Gemini API — технически убрать нельзя.
  Но UI никак не сигнализирует, что это «нормально» или «плохо».

Решение (UI):
  В отображении response_ms добавить цветовую индикацию в history-карточки:
    < 3000 мс  → зелёный текст (#22c55e)
    3000–6000  → жёлтый (#eab308)
    > 6000 мс  → красный (#ef4444)
  Это не решает проблему скорости, но даёт пользователю ориентир.

  В static/js/views/keys.js в функции renderHistoryCard (или аналоге):
    const ms = item.response_ms;
    const msColor = ms < 3000 ? '#22c55e' : ms < 6000 ? '#eab308' : '#ef4444';
    // применить к элементу с временем

Проблемы (по скриншотам):
  a) Фон (.bg-wrap, анимированная mesh-сетка) пропадает вертикально —
     не растягивается на всю высоту страницы при малой высоте контента.

     что ломает читабельность.
  e) В «Топ модели» карточки нормальные, но слишком мало горизонтального
     padding внутри на мобиле.

Решения в static/css/main.css:
  a) .bg-wrap { min-height: 100vh; min-height: 100dvh }
     Убедиться что position:fixed или background-attachment не мешают.

  b) .btn-icon { min-width: 30px; flex-shrink: 0 }
     @media(max-width:640px) {
       .key-actions { flex-wrap: wrap; gap: 4px }
       .btn-icon { width: 28px; height: 28px }
     }

  c) Sidebar overlay: при открытом sidebar на мобиле добавить
     ::backdrop или div#sb-overlay (position:fixed,inset:0,z-index:149)
     который при клике вызывает closeSidebar().
     Проверить что в sidebar.js уже есть или добавить.

  d) Таблица моделей: на мобиле скрыть колонку КОНТЕКСТ (менее важная),
     или сделать таблицу прокручиваемой горизонтально:
     @media(max-width:640px) {
       .docs-table { display: block; overflow-x: auto; -webkit-overflow-scrolling: touch }
     }

  e) @media(max-width:640px) {

     }

B2 (был 1.3) — Группировка уведомлений по типам
────────────────────────────────────────
Статус: [ ] НЕ НАЧАТО
Сложность: средняя. Бэкенд + фронт.

БД — миграция 007_notif_kind.sql (создать файл):
  ALTER TABLE user_notifications ADD COLUMN kind TEXT NOT NULL DEFAULT 'system';
  ALTER TABLE user_notifications ADD COLUMN ref_id TEXT;
  CREATE INDEX IF NOT EXISTS idx_user_notif_user_kind ON user_notifications(user_id, kind);

Типы kind:
  'review'  — решение по отзыву (APPROVE / DELETE / FEEDBACK), бан
  'support' — ответ в поддержке
  'system'  — всё остальное (ключи, лимиты, служебное)

Бэкенд:
  В freeapi/repos/notifications.py:
    • create_user_notification(user_id, message, kind='system', ref_id=None)
      → INSERT ... kind, ref_id
    • get_user_notifications(user_id, kind=None, limit=50, offset=0)
      → если kind задан — WHERE kind=?
    • count_unread_notifications(user_id, kind=None)
      → если kind задан — WHERE kind=?
    • mark_all_notifications_read(user_id, kind=None)
      → если kind задан — WHERE kind=?

  В freeapi/blueprints/notifications_bp.py:
    GET /api/notifications?kind=&limit=&offset=
    → ответ: { items: [...], unread: { all: N, review: N, support: N, system: N } }
    POST /api/notifications/read_all?kind=
    → помечает прочитанными только нужный kind (или все если kind не задан)

  Проставить kind везде при вызове create_user_notification:
    agent.py APPROVE/FEEDBACK/DELETE  → kind='review', ref_id=review_id
    reviews_bp.py PUT status (ручной) → kind='review', ref_id=review_id
    support_bp.py при ответе          → kind='support', ref_id=chat_id
    всё остальное                     → kind='system'

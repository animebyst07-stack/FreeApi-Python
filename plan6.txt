plan6.txt — Точная токенизация: внедрение реальных коэффициентов 3.3 / 1.3

=== ИСХОДНЫЕ ДАННЫЕ ===
Измеренные коэффициенты для Telegram-ботов (Gemini-based):
  - 1 русское слово  ≈ 3.3 токена
  - 1 английское слово ≈ 1.3 токена
  - Лимит контекста остаётся ~180 КБ (CONTEXT_LIMIT_KB = 178.0)

Экономия Dual-Account:
  3.3 / 1.3 = 2.538x — русскоязычный запрос «весит» в 2.5x тяжелее английского.
  Процент экономии: (3.3 - 1.3) / 3.3 × 100 ≈ 60.6% меньше токенов при переводе RU→EN.

=== НОВАЯ ФОРМУЛА estimate_tokens (freeapi/memory.py) ===

1. Разбиваем текст на слова по пробельным символам.
2. Для каждого слова:
   - Если слово содержит хотя бы одну кириллическую букву → вес = 3.3 токена
   - Иначе (латиница, цифры, смешанное) → вес = 1.3 токена
3. Пунктуация (знаки, не являющиеся буквами/цифрами/пробелами):
   - Каждый такой символ добавляет 0.3 токена
   (улучшение точности: «привет!» vs «привет» = +0.3 токена)
4. Изображения: 2000 токенов за штуку (без изменений).
5. Итог: int(sum_word_tokens + punct_tokens + image_tokens)

Обратная функция tokens_to_kb тоже обновляется:
  - Старая: tokens * 4 / 1024 (из расчёта 4 байта/токен)
  - Новая: используем реальный байтовый вес токена для Gemini.
    По измерениям: 1 токен ≈ 4 байта (для UTF-8 mixed content).
    Коэффициент оставляем 4 / 1024, т.к. меняем только подсчёт токенов.

=== ПРОВЕРКА ЛОГИКИ DUAL-ACCOUNT ===
Текущий код в routes.py и tg.py не содержит числовых порогов по коэффициентам —
он просто включает/выключает перевод. Комментарий в UI ("экономия ~40%") был неточным.
После обновления UI явно показывает реальный коэффициент 2.5x, что подтверждает
эффективность перевода.

ИНСТРУКЦИИ ПО РАБОТЕ С РЕПО
──────────────────────────────────────────────────────────

1) Клонирование (всегда первым делом в новой сессии):
   GH_TOKEN=<GITHUB_TOKEN_HERE>
   git clone -q https://animebyst07-stack:${GH_TOKEN}@github.com/animebyst07-stack/FreeApi-Python.git /tmp/FPA

2) Пуш файлов — ТОЛЬКО через GitHub Contents API (git commit на стороне
   агента заблокирован). Шаблон node-скрипта:

   const fs=require('fs'), https=require('https');
   const TOKEN='<GITHUB_TOKEN_HERE>';
   const REPO='animebyst07-stack/FreeApi-Python';
   function ghReq(method,path,body){return new Promise((res,rej)=>{
     const data=body?JSON.stringify(body):null;
     const req=https.request({hostname:'api.github.com',path,method,
       headers:{Authorization:'Bearer '+TOKEN,Accept:'application/vnd.github+json',
       'Content-Type':'application/json','User-Agent':'agent',
       ...(data?{'Content-Length':Buffer.byteLength(data)}:{})}},
       r=>{let b='';r.on('data',d=>b+=d);r.on('end',()=>{
         try{res({status:r.statusCode,body:JSON.parse(b)});}catch(e){res({status:r.statusCode,body:b});}});});
     req.on('error',rej); if(data) req.write(data); req.end();});}
   async function pushFile(path,msg){
     const cur=await ghReq('GET','/repos/'+REPO+'/contents/'+path+'?ref=main');
     const sha=cur.body && cur.body.sha; if(!sha){console.log('NO SHA',path);return;}
     const content=fs.readFileSync(path).toString('base64');
     const r=await ghReq('PUT','/repos/'+REPO+'/contents/'+path,
       {message:msg,content,sha,branch:'main'});
     console.log(r.body.commit?'OK '+path+' '+r.body.commit.sha:'ERR '+JSON.stringify(r.body).slice(0,300));}

   ВАЖНО: file → base64 читать прямо в node (`fs.readFileSync(path).toString('base64')`).
   НЕ передавать base64 как аргумент шелла — будет «Argument list too long».

3) Создание НОВОГО файла — тот же PUT, но без поля sha.
   Удаление файла — DELETE с указанием sha.

4) Перед каждым PUT'ом делать sanity-check:
   • Python — `node -e "console.log('lines',fs.readFileSync('f.py','utf8').split(chr(10)).length)"`
   • Балансы JS-скобок в index.html: count('{') === count('}')
   • Грубая проверка наличия добавленных меток/функций через grep.

5) Один логический шаг = один коммит = один PUT (или несколько, если
   шаг затрагивает 2-3 связанных файла). НЕ пушить полусырое состояние.

ГЛОБАЛЬНЫЕ ПРАВИЛА СТИЛЯ (применяются ВЕЗДЕ)
──────────────────────────────────────────────────────────

• Сайт держится в единой ЧЁРНО-БЕЛОЙ стилистике (оттенки чёрного,
  белого, серого + уже существующий лиловый акцент).
• НИКАКИХ эмодзи нигде: ни в UI-кнопках, ни в табах, ни в сообщениях
  user_notification, ни в push-payload, ни в toast/ошибках, видимых
  пользователю. У эмодзи свой неконтролируемый цвет.
• Все иконки — встроенный inline SVG со stroke="currentColor" /
  fill="currentColor", чтобы цвет наследовался от темы.
• Цветовые исключения допустимы только для НЕкритичных мелких
  элементов (бейдж непрочитанных, точка «новое») — лиловый акцент,
  уже существующий в коде. Новых цветов не вводить.
• Различие между типами/состояниями — РАЗНЫЕ SVG-иконки + opacity/
  толщина обводки, а не разные цвета.
• Тексты — строгие, без воскликов и смайлов. Пример: «Владелец ответил»,
  а не «👑 Владелец ответил!».
• При редактировании существующих сообщений с эмодзи — заменить на
  нейтральный префикс или убрать.
• Логирование на фронте — `clog('TAG', 'msg')` (тег заглавными). На
  бэке — `logger.info('[TAG] msg')`. Без эмодзи.

  send_push_to_user(user_id, title, body, url='/', tag=None):
    payload = json.dumps({'title':title,'body':body,'url':url,'tag':tag})
    for sub in repo.get_push_subscriptions(user_id):
      try:
        webpush(subscription_info={'endpoint':sub.endpoint,
                'keys':{'p256dh':sub.p256dh,'auth':sub.auth}},
                data=payload,
                vapid_private_key=VAPID_PRIVATE,
                vapid_claims={'sub':VAPID_SUBJECT})
        repo.update_push_last_seen(...)
      except WebPushException as ex:
        if ex.response.status_code in (404,410):
          repo.delete_push_subscription_by_endpoint(...)
        else:
          repo.increment_push_failed(...)
        logger.warning('[PUSH] fail status=%s', ex.response.status_code)

    «Выбрано: N → [Одобрить] [Удалить] [Сброс]».
  • Все действия идут одним POST на /api/admin/reviews/bulk.
  • После — обновление списка и тоаст «Обработано: N».

Push: коммит «feat(admin): массовые операции с отзывами».

Эти идеи рассматривались и отвергнуты как преждевременные:

• Command Palette (Ctrl+K) — фича уровня Linear, но в проекте всего
  ~7 view, реальной пользы мало. Имеет смысл когда страниц 30+.
• Heat-map активности (GitHub-style) — декорация, не решает проблем.
• Split-view чат с двумя моделями — много кода, тестовый чат
  используется редко.
• PNG-шеринг отзыва — органический трафик при текущем размере = 0.
• Live Pulse в шапке — добавляет нагрузку, нулевая функциональная
  ценность.
• Trust Level / Reputation — геймификация при десятке юзеров избыточна.
• Similarity-антиспам — пока спама нет, преждевременная оптимизация.
